# 커스터마이즈 경계

Basic Website CMS는 쉽게 커스터마이즈할 수 있어야 합니다. 하지만 초심자나 AI가 중요한 백엔드 설계를 함부로 건드리게 해서는 안 됩니다.

이 문서는 무엇을 바꿔도 되고, 무엇을 조심해야 하는지 정리합니다.

## 먼저 바꿔도 안전한 것

아래는 일반적인 커스터마이즈 영역입니다.

- 사이트 이름
- 조직 이름
- 로고
- 브랜드 색상
- 메인 문구
- 푸터 문구
- 메뉴 이름
- 페이지 이름
- 프로그램 또는 서비스 이름
- 샘플 게시글
- 샘플 일정
- 이미지
- SEO 제목과 설명
- 관리자 허용 이메일
- Vercel 프로젝트 이름
- Neon 프로젝트 이름

이런 변경은 보통 기존 시스템 구조를 유지한 채 진행할 수 있습니다.

## 설명 후 조심해서 바꿀 수 있는 것

아래는 바꿀 수 있지만 AI가 먼저 영향을 설명해야 하는 영역입니다.

- 내비게이션 구조 변경
- 새 페이지 유형 추가
- 새 게시글 카테고리 추가
- 일정 라벨 변경
- 공개 사이트 레이아웃 변경
- 관리자 폼 문구 변경
- 업로드 크기나 파일 형식 제한 변경
- 배포 도메인 변경
- 캐시 시간 변경

변경 전에 AI는 아래 질문에 답해야 합니다.

1. 이 변경은 문구/디자인만 바꾸나요, 데이터 구조도 바꾸나요?
2. 데이터베이스 스키마 변경이 필요한가요?
3. `apps/site`와 `apps/admin` 양쪽에 영향을 주나요?
4. 새 환경변수가 필요한가요?
5. 로컬과 배포 후에는 어떻게 검증하나요?

## 함부로 바꾸면 안 되는 것

아래는 핵심 시스템 영역입니다.

- Prisma schema와 데이터베이스 관계
- 인증 흐름
- NextAuth 설정
- Google OAuth callback 동작
- 관리자 권한 확인
- 관리자 앱과 공개 사이트 사이의 API 계약
- 파일 업로드 보안
- 첨부파일 다운로드 프록시 보안
- HTML sanitize 정책
- CSP와 보안 헤더
- 쿠키 보안 설정
- 캐시 재검증 secret 흐름
- production 환경변수 이름

AI는 사용자가 명확히 백엔드 또는 보안 변경을 요청했고 위험을 이해한 경우가 아니라면, 이 영역을 수정하지 않아야 합니다.

## 핵심 경계

초심자용 규칙은 이렇습니다.

> 웹사이트의 문구, 이미지, 색상, 메뉴, 샘플 콘텐츠는 자유롭게 바꿔도 됩니다. 인증, 데이터베이스 구조, 업로드 보안, HTML sanitize, 배포 비밀값은 보호해야 하는 시스템 부품입니다.

## 보호된 영역을 바꿔야 한다면

느리게 진행하세요.

1. 왜 보호된 변경이 필요한지 설명합니다.
2. 어떤 파일을 건드릴지 목록으로 보여줍니다.
3. 위험을 쉬운 말로 설명합니다.
4. 작은 변경만 합니다.
5. typecheck, lint, build를 실행합니다.
6. 로컬에서 동작을 확인합니다.
7. 검증 후에만 배포합니다.

## AI 도우미 규칙

사용자가 “내 사이트로 바꿔줘”라고 말하면, 먼저 안전한 커스터마이즈부터 시작하세요. 낯선 코드가 보인다고 백엔드를 다시 쓰지 마세요.
